# Anti-Slop Rules

Rules to prevent AI-generated artifacts, unnecessary self-references, and
terminal-hostile Unicode characters.

## File Naming

**NEVER** include "Claude" or "AI" in file names:

```
BAD: claude-helper.ts, ai-generated-utils.ts, AIAssistant.swift
GOOD: helper.ts, utils.ts, Assistant.swift
```

**Exception**: Documentation specifically about Claude (like this rules file).

## Commit Messages

**NEVER** mention "Claude" or "AI" in commit messages:

```
BAD: "Claude implemented user auth"
BAD: "AI-assisted refactor of database layer"

GOOD: "feat: implement user authentication"
GOOD: "refactor: simplify database layer"
```

## Code Comments

**NEVER** add comments that reference AI generation:

```
BAD: // Claude generated this function
BAD: // AI-assisted implementation

GOOD: // Validates user input before processing
GOOD: // See RFC-1234 for implementation details
```

## Documentation

**NEVER** mention Claude in README, docs, or user-facing content unless
specifically documenting AI tooling:

```
BAD: "This component was generated by Claude"
GOOD: "This component handles user authentication"
```

## Pull Requests

**NEVER** mention Claude in PR titles or descriptions:

```
BAD: PR Title: "Claude: Add user dashboard"
GOOD: PR Title: "feat: add user dashboard"
```

## Unicode Characters

**NEVER** use these Unicode characters in code, commits, docs, or output.
They render as garbled sequences in many terminals and violate the principle
of universal readability.

| Banned Character | Codepoint | Use Instead |
|------------------|-----------|-------------|
| Em dash          | U+2014    | -- (two hyphens) |
| En dash          | U+2013    | - (hyphen) |
| Left double quote  | U+201C | " (straight quote) |
| Right double quote | U+201D | " (straight quote) |
| Left single quote  | U+2018 | ' (straight quote) |
| Right single quote | U+2019 | ' (straight quote) |
| Ellipsis         | U+2026    | ... (three dots) |

**Check before committing:** If your editor auto-converts quotes or dashes
to "smart" variants, disable that feature or search-and-replace before commit.

## Why These Rules Matter

1. **Professionalism**: Code should stand on its own merit
2. **Maintainability**: Future developers don't need to know origin
3. **Review focus**: Reviewers should evaluate code quality, not authorship
4. **Clean history**: Git history should describe what changed, not who wrote it
5. **Terminal safety**: Output must render correctly in all terminal emulators
